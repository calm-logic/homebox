# Making a Project Homebox-Ready

Homebox works with any git repo out of the box — there's nothing you have to add for it to deploy. On adopt (and every sync), Homebox clones the repo and dissects it, in order:

1. **`docker-compose.yml` scan** (if one exists) — backing services (Postgres, Redis, …) are reused as-is, and any app service declared there (with a `build:` or a web-ish image) is picked up too.
2. **Build detection** — a `Dockerfile`, or a language marker (`package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `Gemfile`, `pom.xml`, `requirements.txt`, …) at the repo root or in a conventional sub-app directory (`server/`, `api/`, `ui/`, `frontend/`, `backend/`, …) is built with [Nixpacks](https://nixpacks.com) (or the `Dockerfile` directly).
3. **Fallback** — if nothing above matches anything at all, Homebox still attempts a generic Nixpacks build from the repo root rather than giving up.

Traefik labels, the `traefik-net` network, per-environment hostnames, and connection env vars (`DATABASE_URL`, `REDIS_URL`, …) are all generated by Homebox itself at deploy time — you never write Traefik labels or wire networking by hand.

## When you might still want a `docker-compose.yml`

Not required, but it's the natural place for backing services Homebox should reuse:

```yaml
services:
  app:
    build: .        # optional — Homebox will build it from source either way

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

No Traefik labels, no `traefik-net` network declaration, and no host ports — Homebox adds all of that automatically. Backing services just need to be reachable on the compose network; Homebox wires `DATABASE_URL`/`REDIS_URL` into your app for you.

## Overriding detection with `homebox.yaml` (optional)

For anything auto-detection gets wrong, or when you want explicit control (build type, port, subdomain, public exposure, env vars), drop a `homebox.yaml` at the repo root:

```yaml
name: infinitescroll          # optional; defaults to the project slug
services:
  web:
    build: static              # build the SPA and serve the static output
    build_command: npm run build
    static_dir: dist
    public: true                # bare host: <project>.<domain>
  api:
    build: nixpacks              # auto-detect the build (Rust here)
    dir: server
    port: 8080
    public: true
    subdomain: api                # <project>-api.<domain>
    depends_on: [postgres, redis]
  # postgres/redis are still picked up from docker-compose.yml — no need to
  # redeclare them unless you want to override something.
```

A manifest service overrides whatever was auto-detected for that name; anything you leave out is still auto-detected. Supported fields: `build` (`dockerfile` | `nixpacks` | `image` | `static`), `dir`, `dockerfile`, `image`, `static_dir`, `build_command`, `build_image`, `command`, `port`, `public`, `subdomain`, `path_prefix`, `env`, `depends_on`.
